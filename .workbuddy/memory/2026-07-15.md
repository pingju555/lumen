## P3 切页卡死 Bug 修复（用户桌面验收反馈）
- **现象**：两个页面（主页有部件 / 画布为空），切到页面二后**再也无法切回页面一**（热键 `Ctrl+Alt+←` 与底部圆点点击均失效）。
- **根因**：`OnPageChanged` 用 `_switching` 状态位 + 异步双段 Storyboard（淡出→`outSb.Completed` 内重组→淡入→`inSb.Completed` 内 `_switching=false`）。一旦动画 `Completed` 未触发（或重组异常），`_switching` 永久为 true，后续所有 `SwitchTo` 在 `OnPageChanged` 入口 early-return——模型 `Current` 已变但视图不跟随，切页彻底卡死。
- **修复**（`OverlayWindow.xaml.cs`）：移除 `_switching`；`OnPageChanged` 改为**先同步 `ComposeCurrentPage()`（状态永远正确、try/catch 容错），再做纯装饰淡入**（保留 `_fadeSb` 强引用防局部 Storyboard 被 GC 导致页面卡透明）。切页不再依赖动画完成，快速连切也安全。
- build 0 警告 0 错误。用户重新运行 `bin/Release/net8.0-windows/overlay.exe` 验收 →/← 与底部圆点双向切换。
- **经验**：WPF 切页/转场切勿用「状态位 + 异步动画回调重置」模式——回调不触发即永久卡死；正确做法是同步提交状态 + 动画仅作装饰。

## 修复：拖拽行为与属性编辑不对应（双向打通位置定位，2026-07-15 晚）
用户反馈"拖拽行为和属性编辑应该是对应的"。**根因：位置有两条互不通的路径**——拖拽/缩放只改像素 `Bounds`（`MakeDraggable` 的 move/resize DragDelta），但原子位置的**持久属性**是九宫格 `anchor + offsetX/offsetY`（`Atom.RecalcPosition` 每次 `ComposeCurrentPage` 用 offset 覆盖 Bounds.X/Y）。于是：
  - **拖拽→属性 断**：拖完不回写 offset → 保存后 `RecalcPosition` 用旧 offset 把原子**弹回原位**，属性编辑器也显示旧偏移值。
  - **属性→拖拽 断**：属性编辑改 offset/anchor 后 `onPreview` 只调 `atom.Update()`（增量视觉重算，**不重定位**）→ 改了偏移值原子纹丝不动。
- **修复（双向）**：
  1. `Render/Coord.cs` 加静态 `AreaW/AreaH`（当前工作区尺寸）；`OverlayWindow.ComposeCurrentPage` 在 RecalcPosition 前注入 `Coord.AreaW=Width; Coord.AreaH=Height`。
  2. `Atom.WriteBackOffsetFromBounds()`（RecalcPosition 的逆运算）：按当前 anchor 求 base 点，`offset = Bounds.XY - base`，回写 `OffsetXProp/OffsetYProp`（整数像素）。**仅当 offset 为 `PropMode.Static` 时回写**（尊重公式/变量绑定的位置，不覆盖）。
  3. `MakeDraggable`：move 与 resize 的 `DragCompleted` 里先 `WriteBackOffsetFromBounds()` 再 `OnChanged()` → 拖拽/缩放结果落到 offset，重排不弹回、属性编辑器显示一致。
  4. `EditAtom` 注入的 `onPreview` 改为 `() => { atom.RecalcPosition(Width, Height); atom.Update(); }` → 属性编辑改 anchor/offset 原子实时移动。
- 副作用说明：宽高（Bounds.W/H）本就持久化（ConfigStore RectDto）+ Clone 复制，缩放后保存生效、RecalcPosition 仅改位置保留尺寸，故尺寸一直对应；本次只补位置双向。build 0 警告 0 错误。

## 修复：输入框焦点不对 / 无法输入（Popup→Window 根因定案，2026-07-15 晚）
用户反馈"输入框焦点还是不对，无法完成输入行为"。此前两次围绕 `FormulaTextBox` 双层透明、NOACTIVATE 父窗做修复均无效——**真正根因在容器本身**：
- **WPF `Popup` 的 HWND 自带 `WS_EX_NOACTIVATE`**（这是 Popup 设计上"不打断宿主焦点"的意图）。因此即便我们从覆盖层父窗 `RemoveExStyle(WS_EX_NOACTIVATE)`，Popup 自己的 HWND 仍拒绝激活 → 其内部的 `TextBox` 永远拿不到键盘焦点 → 表现为"焦点不对、无法输入"。之前所有"EnterModal 去 NOACTIVATE"方案都打不中这个独立 HWND。
- **修复（根治）**：`OverlayWindow` 的 `EditAtom`/`EditBackground`/`ManageVariables` 三个编辑器**全部从 `Popup` 改为真正的 `Window`**（`CreateEditorWindow(title)` 工厂：`WindowStyle=None` + `AllowsTransparency=true` + `ShowInTaskbar=false` + `Topmost=true` + `SizeToContent=WidthAndHeight` + `MinWidth=280` + 深色 `Background/BorderBrush`）。真实 Window 不带 NOACTIVATE，TextBox 焦点正常。
  - 定位：属性编辑器 `Loaded` 时按 `atom.RootElement.PointToScreen` 锚到原子右侧（超出屏则翻左、夹到 `SystemParameters.VirtualScreen*` 内）；背景/变量编辑器 `CenterWindow` 居中到 `SystemParameters.WorkArea`。
  - 打开即自动聚焦首个可输入控件（`FocusManager.SetFocusedElement` + `win.MoveFocus(First)`），输入立即可用。
  - 关闭回调由 `popup.IsOpen=false` 改为 `win.Close()`；`EnterModal/ExitModal`（去/复 NOACTIVATE 引用计数）保留无害，嵌套（属性内开变量管理器）仍正确。
  - 删除三个 `Popup` 字段（`_editorPopup/_bgPopup/_gvPopup`）；`PropertyEditorPanel`/`BackgroundEditorPanel`/`GvManagerPanel` 三个 UserControl 不变，仅其宿主由 Popup 换成 Window。
- build 0 警告 0 错误，唯一正确构建 `bin/Release/net8.0-windows10.0.22621.0/overlay.exe`。
- **可复用结论（重要）**：在 `WS_EX_NOACTIVATE` 覆盖层上承载"需要键盘输入的浮层"时，**绝不能用 WPF `Popup`**（自带 NOACTIVATE 致 TextBox 无法聚焦）；应改用独立真实 `Window`（或把编辑器内联进主窗视觉树）。这是本项目反复踩坑的核心教训。

触发器条件主要靠比较（`bi(level) < 20`、`mi(playing) = "Playing"`）。排查发现 **`Formula/Parser.cs` 语法层级缺「比较层」**：原层级 expr→logic(&|)→add→mul→unary→primary，**完全没有 < > = >= <= != ~= 的解析生产式**。lexer 已能产出这些 `Op` token、`BinaryNode.ApplyBinary` 也已实现对它们的求值分支，但解析器从不消费它们 → 所有含比较的公式（含文档/FunctionCatalog 大量引用的 `if(bi(level)>20, 高, 低)`）都会解析失败、返回 Null/报错。触发器核心用例形同虚设。
- **修复**：新增 `ParseCompare()` 生产式（consumes `> < >= <= = != ~=`），插入 logic 与 add 之间；`ParseLogic` 改为调 `ParseCompare`（而非 `ParseAdd`），保持优先级 `&|` 松于比较、比较松于算术。等号用单字符 `=`（与 KLWP 一致）；`==` 非合法 token（会被拆两个 `=`，须避免，已同步修正触发器 Tooltip 示例 `mu() == "playing"`→`mi(playing) = "Playing"`）。
- **验证**：建临时独立测试工程（引用 overlay 工程）跑 `FormulaEngine.Eval` 8 例——`3 < 5`=True、`bi(level) < 20`=True、`10 >= 10`=True、`mi(playing) = "Playing"`=False(无媒体)、`5 = 5`=True、`3 > 10`=False、`if(2 < 3,100,200)`=100（此前 if 内比较会失败，现已通）、`1 ~= 2`=False。全部正确。测试工程已删除。
- **影响面**：此修复顺带修好全系统所有含比较的公式（此前均为潜藏 bug，因无 GUI 验收未暴露）。build 0 警告 0 错误。
