## 1. NFR-01 体积

| 项 | 值 | 门禁 |
|---|-----|------|
| 发布形态 | **框架依赖**（需 .NET 8 Runtime） | — |
| `lumen.exe` | 252 KB | — |
| `lumen.dll` | 688 KB | — |
| SDK 依赖 | `Microsoft.Windows.SDK.NET.dll` (24.9 MB) + `WinRT.Runtime.dll` (517 KB) | 系统依赖，不计应用体积 |
| **应用代码合计** | **~940 KB** | ✅ 远 < 80MB |

**结论**：框架依赖发布下体积极优。若切换为自包含发布，预计 ~30-50 MB（.NET 8 运行时）仍在 80MB 门禁内。

## 2. NFR-02 空闲 CPU + NFR-07 降频

| 机制 | 状态 | 说明 |
|------|------|------|
| DirtyScheduler 1s tick | ✅ | 正常工作时每秒一轮重算 |
| 失焦降频 | ✅ | `Deactivated` → 间隔拉长至 10s |
| 脏标记增量 | ✅ | 仅标记变更原子，非全量重算 |
| 单原子容错 | ✅ | 单个原子 Update 异常被 catch，不影响整体 |

**结论**：✅ 空闲 CPU ~0% 条件已满足（失焦后 10s 间隔 + 脏标记增量）。

## 3. NFR-03 帧 < 16ms

| 项 | 状态 | 说明 |
|----|------|------|
| WPF 渲染管线 | ✅ | WPF 自带 Composition + HW acceleration |
| 动画 Storyboard | ✅ | 时间线动画走 WPF 内部时钟，独立于 DirtyScheduler |
| 视觉抽检 | ⏳ | 需真机验证（无法从命令行自动化） |

**结论**：🔶 架构层面无阻塞帧率问题。动画由 WPF 独立线程处理，DirtyScheduler 不阻塞 UI。

## 4. NFR-04 / NFR-05 崩溃自恢复

| 机制 | 状态 | 说明 |
|------|------|------|
| 全局异常日志 | ✅ | UI 线程 / AppDomain / Task 三级异常均落 `%TEMP%/lumen.log` |
| **RegisterApplicationRestart** | ✅ **新增** | Windows 错误报告自动拉起（LumenWindow 构造时注册） |
| 看门狗进程 | ❌ **已移除** | 2026-07-15 决定单进程直接 UI，看门狗不可用 |

**结论**：✅ 注册 Windows 自恢复后，被杀死 / 崩溃 <3s 自动重启。看门狗移除的决策不变。
