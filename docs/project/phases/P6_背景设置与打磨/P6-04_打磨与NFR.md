# P6-04 打磨与 NFR

> Phase：P6 · 背景 + 设置面板 + 持久化闭环 + 打磨
> 上游：`v1开发计划.md` P6（打磨）、`功能需求.md` §8 非功能（NFR-01~07）、`技术栈选型.md` §4（打包）/ §5（测试）
> 关联：`P0-02_守护进程.md`（自恢复）、`P2-05_增量重算与容错.md`（降频 / CPU）、`P6-03_持久化闭环.md`

## 目标
达到非功能门禁：
- **NFR-01 体积**：Release exe < 80MB（或按 `技术栈选型.md` §4 取舍放宽）；自包含裁剪 vs 框架依赖二选一。
- **NFR-02 空闲 CPU ~0**：`P2-05` 降频 + 失焦停绘生效。
- **NFR-03 帧 < 16ms**：渲染 / 动画不卡。
- **NFR-04 崩溃自恢复**：`P0-02` 看门狗 + `RegisterApplicationRestart`。
- **NFR-05 被杀自恢复**：看门狗重拉。
- **NFR-06 内存**：常驻达标（无泄漏）。
- **NFR-07 降频**：见 `P2-05`。

## 范围
**包含**
- `dotnet publish` 体积核对：框架依赖（A）/ 自包含裁剪（B）二选一，记录产物大小。
- 性能抽检：空闲 CPU、帧时间、内存（任务管理器 / `dotnet-counters` / `PerfView`）。
- 恢复验证：杀进程 / 抛异常崩溃 / 重启均自恢复。
- 单屏 → 单实例验证（v1 锁单屏，多实例留 v2）。
- 每 Phase 结尾已做体积 / 内存抽检（`v1开发计划.md` 备注），此处收口。

**不含**：多屏多实例（v2）；深度性能剖析工具化（v1 手工抽检 + 关键计数器）。

## 关键设计
### 打包取舍（§4）
```text
A. 框架依赖发布：体积小（~数 MB），需目标机有 .NET 8 Runtime。
B. 自包含裁剪：体积大（~50–70MB 裁剪后），免 Runtime，守 <80MB 关键。
v1 建议 B（自包含裁剪）以"双击即跑"，严守 <80MB；若超则评估 A 或再裁剪。
```

### 抽检清单
```text
[ ] dotnet publish -c Release -r win-x64 --self-contained true / -p:PublishTrimmed=true
[ ] 核对 overlay.exe + 依赖 < 80MB
[ ] 空闲 5min：CPU ~0%（任务管理器）
[ ] 拖动原子：帧 < 16ms（视觉流畅）
[ ] 杀 UI 进程 → 看门狗 <3s 拉起
[ ] UI 抛未捕获异常 → 系统 RegisterApplicationRestart 重启
[ ] 常驻 1h：内存无持续增长
```

## 技术选型
`dotnet publish`（框架依赖 A / 自包含裁剪 B）、`dotnet-counters` / `PerfView`（性能）、xUnit 单测（公式引擎 / 数据接入，`技术栈选型.md` §5 测试）。无额外依赖。

## FR 映射
NFR-01~07（`功能需求.md` §8）；`v1开发计划.md` P6 打磨、`技术栈选型.md` §4 打包。

## 验收
Release exe 体积达标；空闲 CPU~0；帧 <16ms；杀 / 崩均自恢复；常驻内存稳定；单实例单屏验证通过。

## 依赖与顺序
- 依赖：全 Phase（最终收口）；`P0-02`(恢复)、`P2-05`(降频)、`P6-03`(配置大小)。
- 被依赖：版本发布门禁（`版本排期.md` §6 跨版本横切 NFR 门禁）。

## 风险 / 开放
- 体积取舍（§4）需在 v1.0 早期定，影响发布形态；NFR-01 <80MB 在自包含下偏紧，可能需裁剪 / 或放宽（开放决策见 `版本排期.md` §8）。
- 自包含裁剪可能误剪 WinRT 投影（SMTC/PackageManager，P4）所需类型 → 需显式 `TrimmerRootDescriptors` 保留。
- 性能抽检需真机，CI 难自动化，v1 以手工为主。
