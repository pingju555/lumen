# 技术栈选型（Desktop Overlay Layer）

> 状态：**v2 定稿（2026-07-14 修订）**。决策变更：由 Rust + Tauri + Slint 切换至 **C# / WPF (.NET 8)**。
> 约束沿用：仅 Windows、内存尽量小、原生覆盖层（无 WebView/网页部件）、后台防误杀。
> 切换主因：覆盖层需直接与 Win32 窗口层级 / 样式 / 点击穿透打交道；Tauri + Slint 的框架抽象与 Win32 定位反复冲突，编译-修复循环过多。WPF 原生持有 `HWND`，XAML 直接描述原子 / 层，开发可控性更优。

## 0. 修订说明（v1 → v2 切换）

- **原栈**：Rust（`x86_64-pc-windows-msvc`）+ Tauri v2 + Slint 1.x（零 WebView2，已用 `tauri-slint-noview-poc` 技能验证可编译出原生 exe）。
- **新栈**：C# + WPF（.NET 8.0.421 SDK 已装，VS2022 Community 可编译）。
- **切换理由**：覆盖窗口层级 / 无边框 / 输入拦截 / 任务栏 Auto-hide 铺满均需 Win32 直控；Slint 窗口与 Win32 定位模型打架，报错循环拖慢进度。WPF 本身即 Win32 窗口 + XAML 声明式 UI，无需框架抽象层。
- **沿用**：公式引擎（纯逻辑，重写为 C#）、双模式互相看门、JSON 持久化、全本地数据源。
- **待确认**：NFR-01（<80MB）在 WPF 自包含发布下可能需放宽，见 §4。
- **2026-07-15 补充**：新增 §5 横切技术选型（NuGet 白名单 / deferred 分支 / 工程实践）+ 扩展 §6 模块矩阵至覆盖 `功能需求.md` 全部 FR 组（显隐/生命周期/托盘/图标字体/图表/文件速览/多屏/DPI/日志/配置校验/DI 等），实现全量代码级选型。

## 2. 为什么是 WPF

覆盖层需要三件"系统级"能力，WPF + Win32 互操作可直控，无需框架封装：

- **窗口层级与样式**：`Window` 即 Win32 窗口，原生持有 `HWND`，可直接 `SetWindowLong` 改扩展样式（无边框、输入拦截 / 方案 B 透明穿透），用 `SetWindowPos` 设层级（盖桌面图标、居普通应用之下）。Tauri/Slint 阶段这些被框架抽象挡住，正是报错源头。
- **任务栏不预留（Auto-hide）**：任务栏由**系统自动隐藏（Auto-hide）**，覆盖层直接铺 `SystemParameters.WorkArea`（等价于全屏，不含永久任务栏条），**不预留**固定任务栏区；任务栏弹出时浮于覆盖层之上（局部遮挡可接受）。省去原 `ABM_GETTASKBARPOS`/`SHAppBarMessage` 手写预留逻辑。
- **托盘 / 单实例 / 全局快捷键**：`Hardcodet.NotifyIcon.Wpf`（托盘）成熟；单实例用命名 `Mutex`；全局快捷键用 `RegisterHotKey` P/Invoke。

> XAML 负责原子 / 层 / 画布 / 设置面板的声明式渲染（含动画 `Storyboard`）；Win32 P/Invoke 负责覆盖窗口的定位与穿透；公式引擎为纯 C# 逻辑（无 UI 依赖，可单测）。

## 3. 守护进程设计（防误杀）

### 3.1 形态：单一 exe 双模式
- `overlay.exe` —— 正常启动 → 覆盖层 UI（WPF `Application` + `MainWindow`）。
- `overlay.exe --guardian` —— 无窗口看门狗进程（不初始化 WPF，纯 C# 监控循环）。
- 分发 = **一个文件**。自启时主程序 `Process.Start` 拉起 `--guardian` 兄弟进程。

### 3.2 互相看门（mutual watchdog）
两个独立 OS 进程，互为兄弟，互相持有对方 `Process` 句柄并等待退出：

```
overlay.exe (UI)
   │ 启动即 Process.Start("overlay.exe --guardian")
   ▼
overlay.exe --guardian (看门狗)
   ├─ 轮询 WaitForExit / 命名 EventWaitHandle
   │     → UI 异常退出（非 clean）⇒ 立即重启 UI
   └─ 写心跳到命名管道 \\.\pipe\overlay-guardian

overlay.exe (UI)
   └─ 监听管道心跳 / WaitForExit(guardian)
         → 看门狗被杀 ⇒ 立即重启 --guardian
```

- **心跳通道**：命名管道 `\\.\pipe\overlay-guardian`，看门狗每 ~3s 写 tick；UI 侧 ~6s 无 tick 即判看门狗死亡并重拉。
- **Clean 关闭信号**：命名事件 `Global\overlay-clean-shutdown`。用户主动退出（托盘"退出"）时先置位该事件，再退出；对方检测到事件置位 → **不重启**（正常退出）。
- **防误杀覆盖场景**：误点关闭、崩溃、任务管理器逐个结束其中一个 → 另一个在秒级内拉起。本机管理员同时结束两者不在防护范围（用户明确：只需防误杀）。

### 3.3 崩溃兜底
- 看门狗独立进程：UI 崩溃由看门狗 `WaitForExit` 捕获并重拉，形成闭环。
- 看门狗自身崩溃由 UI 的 `Process` 等待捕获并重拉。
- （可选）`RegisterApplicationRestart` 作为二次保险。

### 3.4 开机自启
- 写入注册表 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` → 用户登录即启动 `overlay.exe`（随之拉起看门狗）。
- 零安装、可一键取消（删除注册表项）。

## 4. 构建与产物

- 工具链：.NET 8 SDK（`dotnet build`/`publish`），MSVC 运行时（VS2022 自带）。
- 命令：`dotnet publish -c Release -r win-x64` → `bin/Release/net8.0-windows/win-x64/publish/overlay.exe`。
- 体积策略（NFR-01 待确认取舍）：
  - **方案 A（推荐起步）**：框架依赖发布（framework-dependent），运行时已由 .NET 8 SDK 提供，产物极小（几 MB），但目标机需装 .NET 8 运行时。
  - **方案 B**：自包含 + 裁剪（self-contained + trim），无需预装运行时，但 WPF 全量裁剪受限，体积约 90–140MB，**可能突破 <80MB**。
  - 取舍：若坚持 <80MB，走 A（前提用户机已有 .NET 8）；否则放宽 NFR-01 走 B。需用户拍板。
- 打包：本期直接分发；后续可加 MSIX / InnoSetup（写注册表自启、建开始菜单项）。

### 5.2 明确 deferred 分支（v1 不纳入，但锁定技术路径）
- **网页部件（FR-WEB / §6.5）**：v1 坚持无 WebView。若未来确定纳入 → 用 **WPF `WebView2` 控件**（`Microsoft.UI.WebView2`），**不切 Tauri/Electron**（避免双栈与现有 WPF 架构冲突）。
- **本地 LLM（FR-LLM / §6.6）**：v1 不纳入。若做 → **原生 `HttpClient` 调本地服务 + 原生 XAML 对话 UI**，避免 WebView。
- **天气外部数据（FR-CW-06~09）**：v1 不接入外部 API（全本地决策）。待用户拍板数据源/Key 后再选 HTTP 客户端（届时即 `HttpClient`）。
- **动态壁纸/视频（FR-BG-06）**：v1 不做。
- **多实例多屏（FR-COV-06/07）**：v1 单屏单实例；schema 已支持，技术路径 = 单进程多 `Window` 按屏（每屏一个覆盖窗口），架构细化时定。

### 5.3 工程实践
- 单测：**xUnit**，`dotnet test`；公式引擎（纯逻辑）为单测重点，目标覆盖 19 函数 + 求值边界。
- 日志：结构化 `ILogger` → 调试期 `Debug`/文件 `%TEMP%/overlay_p0.log`；守护进程崩溃兜底写同日志。
- 崩溃安全（NFR-05）：覆盖层隐藏即释放输入（FR-COV-12）；守护进程（§3）保证原生桌面不丢失。

### 6.1 合成 / 混合模式策略（WPF 原生支持）
WPF `UIElement` 原生支持 `Opacity`；混合模式（multiply / screen / overlay）可通过 `UIElement` + 自定义 `ShaderEffect`（`PixelShader`）或 `VisualBrush` 合成实现，无需引入额外渲染面。
- **v1**：Normal + Opacity 直接可用；混合模式用 `ShaderEffect` 实现（WPF 比 Slint 更优，无需 Skia/Direct2D 自定义层）。
- 复杂混合（如 per-pixel 多源合成）可预留 `RenderTargetBitmap` 离屏合成路径（v1.x）。

### 6.2 基于焦点的渲染降频（NFR-07）
WPF 为保留模式（retained-mode），本就仅在视觉树变更时重绘；公式 tick / 动画驱动 `Invalidate`。降频策略：
- **获焦 + 有动态内容**：`DispatcherTimer` 满帧（如 30–60fps 或按需 1s tick 时钟）。
- **失焦**：降低 tick 频率至低帧（如 4fps），维持动态内容（时钟 / 监控）更新但不浪费资源。
- **失焦且无动态活动**（所有公式 tick 空闲、无动画）：停止 `DispatcherTimer`，完全事件驱动刷新（空闲 CPU 近 0，与 NFR-02 协同）。
- **重新获焦**：恢复满帧 / 正常 tick。

### 6.3 系统指标数据源（已定：PDH 直采）
`si()` 需 CPU / 内存 / 网速 / 磁盘 / 进程。
- **✅ v1 采用**：`System.Diagnostics.PerformanceCounter`（底层即 PDH）。网速（每网卡）/ 磁盘吞吐精确；内存 / CPU 直取，优于 `sysinfo` 精度。
- v1.x 可对个别计数器做聚合精化，不影响 v1 架构。
